Image Optimization

2026-04-13

If one can cache cheaply, then cache you should.

But I found out the hard way that without good edge caching or simply that ONE time the user has to load your site for the first time, and you don’t have any optimization, the image size and quantity of them loaded can matter a lot for page load times.

This is a topic I don’t see talked about for simple static sites too often, but I think it can really make a difference to have some nice-looking images that load quickly. It just feels good.

So, I went on a bit of a journey to learn about images and image files. It can get complicated fast. I am not claiming to be an expert, but understanding the gist has helped me recently with minor image optimization.

There are several tools out there for making images smaller, but the simplest way is to just take the image and open it in GIMP or Photoshop, reduce it down to the smallest canvas size and resolution possible with a common aspect ratio, then export it as a .jpeg, or better if you can.

Some larger images sprinkled here and there are usually fine for a modern internet connection, but consider that many people viewing your site may also have a bad connection or use accessibility tools, I don’t want my site to be completely inaccessible to those people.

Size Matters

Ensuring images are the size they appear on screen can help significantly. This simplifies CSS for images and cards and maintains consistent aspect ratios to prevent unnecessary rendering.

If you have to, using SVGs instead of other formats like JPEG allows for better dynamic resizing and a crisper visual. Or at least it's supposed to, depending on your browser settings.

Also, another advantage of images like SVGs is that placing them ‘inline’ also prevents the browser from having to fetch or cache external files.

will fetch:

<img src="/willcall.svg">

Is already there and is rendered without a fetch:

<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 600 600"> <path fill-rule="evenodd" stroke="#fff" stroke-width="6" d="m412 375 57 72 2 5-63 ...."/></svg>

Aspect Ratio

Another point, I typically use a 16:9 aspect ratio just because its so common and very close to many social media platforms and screens that additional optimization is usually not worth the hassle. I then scaled in increments up to 1280x720. Anything larger is usually overkill unless the circumstances are unique.

Again, it's super easy to open the image you want, change the scaling and image size to what you want with a bit of cropping, and export in the appropriate format. Oftentimes, you don’t need to take it any further.

Testing

So see the results, in your browser of choice > right click or settings > inspect > network tab.

In here, I want to visually see load times and sizes of files fetched as I load pages and resources. It's not very scientific, but I can confirm I reduced page load times by almost a second in some cases just by making the images smaller, using some lazy loading, and removing unnecessary CSS.

Compression Algorithms Used

JPEG and PNG image formats are great, but they are a bit dated. If you want to take it a step further, you can ditch that smelly old JPEG for a fancy new algorithm.

The popular new kids on the block are WebP and AV1 Image File Format (AVIF). Despite AVIF having some advantages, it is a bit newer than WebP, so to help with slightly better support, I typically gravitate toward WebP for now and use a PNG or JPEG fallback. If you can't use WebP or AVIF, you could always give MozJPEG or OxiPNG a try. I have heard great things about them.

Note that image compression is not a silver bullet for making images smaller, and depending on how you are using the compression, it might result in worse performance. So always benchmark! Don’t read some dribble like this and randomly go do it on a whim.

Fallback Images

For newer algorithms, some older browsers and tools just don’t support them.

This will tell the browser that you want WebP, but if the person viewing your site is allergic to updating their browser/PC, it will attempt to use the fallback, and in the worst case, use your alt text.

<picture> <source srcset="blah.webp" type="image/webp"> <img src="/assets/img/backup.png" alt="epic image of epicness"></picture>

Keep in mind that if you try to test the fallback manually, just breaking the path to the WebP file will not always invoke the browser to use the fallback image, it's only going to use the fallback if the type is not supported, not if the image literally does not load in some cases. I found this out the hard way. Your mileage may vary.

Lazy Loading

If you have a ton of unavoidable images on the screen, try lazy loading a few of them if it won’t break the visual effect you are going for. Modern browsers support the loading="lazy" attribute. This tells the browser to hold off on the request until the image is near the viewport.

<img src="large-photo.jpg" loading="lazy" alt="Description">
