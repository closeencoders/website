Learning GO as a Noob

2026-03-17

Overview

Learning GO with a heavy Java background has been relatively easy for me so far, but I really want to get into the nitty-gritty. I’m rather agnostic with what I do in code as long as it's fun, so a year from now I might drop GO entirely if I find it annoying, but currently it feels extremely simple and easy to use.

I want to document the hurdles and issues I have run into and some of the key differences in what some of the community would refer to as “Effective GO” or “Best Practices”. Most of these practices are an opinion, but to at least know of them can help with readability for other developers and help alleviate some frustration as you make your own judgments about the language.

Lastly, I am collecting tutorials and resources, so I am assuming others will too. The internet has a plethora of resources, but which ones are relevant?

Starting Out

Unsurprisingly, the most difficult thing for me is not having classes or inheritance and interfaces not being as verbose as Java. Method receivers and pointers were a bit of a dance at first, but not difficult.

It's possible to mimic the feel of OOP in GO, but overdoing it because you want an OOP vibe in a complex code base can become frustrating, and the language may start to fight you.

This does not mean you can’t fore the code to look like its using GOF-inspired patterns if you want, but GO can be annoying if you want several layers of abstraction and immutability and other various things that patterns may rely on in a classic sense. For example, you can use the final keyword in Java on lots of things, GO uses const and is a bit more restrictive with what you can use it on to make it immutable.

Hopefully I will get more into const later, but this was a great video that goes into depth: Backend Banter Podcast.

I want to emphasize there is nothing wrong with making the code look and feel any way you want if it works, but I have personally found it makes it difficult to read and modify, especially if done poorly.

The reason I know this is that I write poor GO code on the side, and I have to refactor it.

Nitty-Gritty

It may be a skill issue, but going in the 100% in the opposite direction of OOP, I find equally as confusing. Ignoring some or all of GO’s features and writing only ‘procedural-ish’ code leads down a similar path of pain.

There are way more things to learn, but if you already know how to program, I think the key is to really get comfortable with three core ideas.

Interfaces

Using interfaces only when you really need them.

In go, you only need to make the code look like an interface to use the interface.

I hated this when I first started, and still find it confusing if I don’t name my files and packages well right out the gate.

While it has some obvious cons, a pro I have found so far is that you are super decoupled from one part of the code base to another, and getting used to it can be very powerful. I have seen some great examples like This Video Explains, but I have not personally run into anything at this scale yet, outside of practice projects.

Packages

The directory/package structure in GO is more important than it seems.

When the code starts to increase in complexity, package names are super important because it's how you pull in other code. Not by class name like Java, but by the relative path. GO quite literally uses a directory named “internal” to explicitly tell the package structure not to expose what is inside it externally to other projects. If you want those features, its a must to understand how it works.

Personally, I still have not mastered interfaces or come up with a package structure I completely agree with. I’m still winging it, but I kinda like that, it's nice that you can do whatever you want if you want.

Structs

Using structs can feel a lot like using classes, but they are NOT classes.

I used structs as a class replacement, and it was not fun, and I was very confused. Don't do that, they have similarities and thats it.

Other Complaints

There are some low hanging fruit complaints that get thrown around a lot, such as error handling, walrus operators, and using capital or lower case characters to declare private vs public access to your code. While valid, others have covered them in great detail.

Resources

There are tons of free and paid resources to learn Go. I would recommend having a clear goal in mind of what you want to do before spamming a textbook that is a few years out of date, as the language has changed a bit over time.

The goal is not to waste your time and learn key concepts first, not an abstract niche concept only to find out a month later it's deprecated or its built into the language now. (I have done this)

Starting With the Source

Go’s webpage has some amazing resources in /learn, the same way many other modern languages and open source tools do.

If you already know how to code, even just a bit, I highly recommend working through “GO by Example” before buying a book or anything else. It is super easy and touches on just about everything you need to get off to a good start.

Next, go build a few super simple projects, you know, your typical TODO app, etc. I personally built a static site generator for this website. The general process wasn’t too complicated, but I learned a lot since I’m not proficient in GO yet.

Influencers

If you are anything like me, you learn by doing and by watching others who are more skilled in their field. Over time, I found a handful of YouTubers and bloggers, some of whom also have podcasts, who use GO regularly and have a much higher level of expertise than I do.

Almost every time I watch their content, I learn something or get a different perspective on how to approach the development.

(Not affiliated and not in any value order)

  • Melkey - An ex-Twitch engineer who often uses and makes tutorials with GO, and talks about the industry as a whole.
  • Mongoose Studios - No AI, no BS streams about his adventures developing GO applications.
  • Avery - I talked about this guy in a previous post. He has a great introduction playlist to Raylib and GO if you are just starting out like me.
  • ThePrimeagen - An ex-Netflix engineer with a larger channel. He often talks about and uses GO for his projects. He has several different channels, not just the one linked.
  • Bob Nystrom - Author of a few GO books and has a great blog.

Tools and Books

There are so many free resources and tools out there. I do not condone pirating books as PDFs, but there are several blatantly free GO books. Just take a look at Book Search on Goodreads to get a feel for what people like and what works.

I am not affiliated with this guy in any way, and it’s not free, but I am personally biased towards Writing an Interpreter in GO.

Another great resource I see so many people reference is roadmap.sh, it is a great place to start if you really need a structured approach to learning just about anything in tech. It can be found in the “GO Roadmap” section.

Best of luck!
