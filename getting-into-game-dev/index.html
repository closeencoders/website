Learning Game Development as a Noob

Exploring the itch to truly understand how games work but not knowing where to go yet. A look at my experiences, mistakes, and plan going forward.

By Josh - Mar 11, 2026 | software, learning

I want to make video games and know how they work at a deeper level.

I don't want to generate code and assets with an AI, or let a modern game engine do all the work, I want to understand what’s happening under the hood to feel like I’m getting my hands dirty before using more advanced tools. My hope is that other people do too, and if someone reads this, maybe they can learn from my mistakes.

In a previous life, I tried to make a few games in Java, but it always felt clunky, like the language was fighting me. Java would scream, “Where is my Jira ticket number and business requirements!?” as I forced it to loop through entities in some static factory thing. I’m sure that for some, Java is a great tool for making games, but it’s just not for me.

I wondered if making games was just not my cup of tea for a while, but the itch never really went away. It just looks so cool and rewarding.

Trying New Things

So I am trying to make the switch to another language that will let me mess around with Raylib and get the hang of the basics before I jump off the deep end.

The only real criteria was to loop through some assets and run update() and render() at 60fps in a simple 2D world for a bit without wasting a bunch of time. That’s all I want and could ever ask for. A bonus point if the language is general-purpose.

So, I took it into my own hands to waste my time instead and looked at C, C++, C#, Zig, Python, Odin, (took a peek at Jai), and even, despite the fact that I want to wait to use power tools GDScript in something like Godot.

I feel like I looked at everything and sucked at it. You name it. I never looked at the Unity engine, though, I hope I don’t regret that.

Landing on GO

While Jai looks epic and writing C made me feel cool, I felt too incompetent to use them effectively (yet). So, after enough time passed, I landed on GO of all things. I really want to give Jai a real try after it’s out of beta. I do enjoy a good Jonathan Blow video or rant every once in a while too.

I originally disliked Go coming from Java. Using pointers and method receivers felt like jumping into a pool of ice water at first. However, as time went on, I kept gravitating back to the bizarrely simple, easy to read syntax, even with its quirks and repetitive not nil error checking.

if err != nil {
    aaaahhhhh!
}

It just does what I need it to do without some weird pattern, straight to an .exe. I don’t need a Gradle build, a Maven repository, or to package a jar thingy in another thingy just because the thingy said so with an annotation that can’t load it into memory because your POJO is sad!

(Sorry, repressed feelings. Java is its own thing with pros and cons like any other language.)

First Raylib + GO Game Tutorial

I wanted a tutorial so simple it was painful. I landed on @jarreed. In seven episodes, I had all the basics set up, Go with Raylib installed, a basic game loop, and some practice sprites to make your character walk around. I really wish he had continued the series. Sadly, there are not too many video Go + Raylib tutorials out there.

Using Raylib and Go has worked really well so far, but I can already see how difficult just a simple 2D game would be without a framework like Raylib or trying to build something even more complex without an out of the box engine like Godot.

I have more respect for the process now. The mistake I made here is going head first into tutorial h311. I broke out of it and started making my own projects on the side to build more real world skill, but that is only because I have been here before.

My Biggest Mistake

If you are in the same position as I am, trying to figure out what to do or how to build a game, my advice is not to worry about the tech stack or programming languages, within reason, of course.

My biggest mistake was bouncing around trying to use lots of different languages to try to make a game,  it was like tutorial hell, but worse. Take your primary language and dive deep into it to grasp the basics of what you’re trying to learn first, to see if you even want to do it.

I now understand and respect how insanely difficult it is to make a larger game, and I wish I had attempted to build more projects and gained that respect and built some additional skill sooner.

Even if you don’t know how to code yet, choose a general-purpose language and/or game engine that seems interesting to you. It matters far less than it seems in the beginning because you have no idea what you are doing anyways.
